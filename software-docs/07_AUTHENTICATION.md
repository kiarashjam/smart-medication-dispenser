## 2. JWT Authentication

### 2.1 Token Generation

JWT tokens are generated by `JwtAuthService` on login or registration:

```csharp
// JwtAuthService.cs
public string GenerateJwt(Guid userId, string email, UserRole role)
{
    var key = new SymmetricSecurityKey(Encoding.UTF8.GetBytes(_settings.SecretKey));
    var creds = new SigningCredentials(key, SecurityAlgorithms.HmacSha256);
    
    var claims = new[]
    {
        new Claim(ClaimTypes.NameIdentifier, userId.ToString()),
        new Claim(ClaimTypes.Email, email),
        new Claim(ClaimTypes.Role, role.ToString())
    };
    
    var token = new JwtSecurityToken(
        issuer: _settings.Issuer,
        audience: _settings.Audience,
        claims: claims,
        expires: DateTime.UtcNow.AddDays(7),
        signingCredentials: creds
    );
    
    return new JwtSecurityTokenHandler().WriteToken(token);
}
```

### 2.2 Token Structure

| Part | Content |
|:-----|:--------|
| **Header** | `{ "alg": "HS256", "typ": "JWT" }` |
| **Payload** | `{ "nameid": "<userId>", "email": "<email>", "role": "<role>", "exp": <timestamp>, "iss": "SmartMedicationDispenser", "aud": "SmartMedicationDispenser" }` |
| **Signature** | HMAC-SHA256 using secret key |

### 2.3 Token Claims

| Claim | Type | Description | Example |
|:------|:-----|:------------|:--------|
| `nameid` | ClaimTypes.NameIdentifier | User ID (GUID) | `a1b2c3d4-...` |
| `email` | ClaimTypes.Email | User email | `patient@demo.com` |
| `role` | ClaimTypes.Role | User role | `Patient`, `Caregiver`, `Admin` |
| `exp` | Standard | Expiration (7 days) | Unix timestamp |
| `iss` | Standard | Issuer | `SmartMedicationDispenser` |
| `aud` | Standard | Audience | `SmartMedicationDispenser` |

### 2.4 Token Validation

Configured in `Program.cs`:

```csharp
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(o =>
    {
        o.TokenValidationParameters = new TokenValidationParameters
        {
            ValidateIssuerSigningKey = true,
            IssuerSigningKey = new SymmetricSecurityKey(Encoding.UTF8.GetBytes(jwtKey)),
            ValidateIssuer = true,
            ValidIssuer = "SmartMedicationDispenser",
            ValidateAudience = true,
            ValidAudience = "SmartMedicationDispenser",
            ValidateLifetime = true,
            ClockSkew = TimeSpan.Zero  // No clock skew tolerance
        };
    });
```

**Validation checks:**
1. Signing key is valid (HS256)
2. Issuer matches configuration
3. Audience matches configuration
4. Token has not expired
5. No clock skew tolerance (strict expiry)

### 2.5 Using JWT in Requests

**Web Portal (Axios + localStorage):**

> **Note:** The web portal authenticates via the real backend API (authApi.login, authApi.register, authApi.me). JWT tokens are stored in localStorage under the key 'token' and attached to all API requests via an Axios interceptor.

```typescript
// api/client.ts — reads from localStorage key "token"
api.interceptors.request.use((config) => {
  const token = localStorage.getItem('token');
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});
```

**Mobile App (AsyncStorage + In-Memory):**
```typescript
// Token persisted in AsyncStorage under key "token"
await AsyncStorage.setItem('token', data.token);

// Also set in-memory for Axios interceptor
export let authToken: string | null = null;
export function setAuthToken(token: string | null) { authToken = token; }

// Request interceptor adds Bearer token from in-memory variable
api.interceptors.request.use((config) => {
  if (authToken) config.headers.Authorization = `Bearer ${authToken}`;
  return config;
});

// On app launch, token is restored from AsyncStorage → setAuthToken()
```

**cURL Example:**
```bash
curl -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIs..." \
     http://localhost:5000/api/devices
```

## 3. Password Hashing

### 3.1 BCrypt Implementation

```csharp
// JwtAuthService.cs
public (string Hash, string Salt)? HashPassword(string password)
{
    var salt = BCryptNet.GenerateSalt();
    var hash = BCryptNet.HashPassword(password, salt);
    return (hash, salt);
}

public bool VerifyPassword(string password, string hash) =>
    BCryptNet.Verify(password, hash);
```

### 3.2 Security Properties

| Property | Value |
|:---------|:------|
| **Algorithm** | BCrypt |
| **Salt** | Auto-generated per password |
| **Work factor** | Default (10 rounds = 1024 iterations) |
| **Storage** | Only hash stored in database (`User.PasswordHash`) |
| **Verification** | Constant-time comparison (BCrypt.Verify) |

## 4. API Key Authentication (X-API-Key)

### 4.1 Overview

API keys are used for machine-to-machine authentication — ESP32 devices and external systems calling webhook/sync endpoints.

### 4.2 API Key Lifecycle

```
1. User creates API key in Web Portal (Integrations page)
   → POST /api/integrations/devices/{deviceId}/api-keys
   → Returns: { apiKeyId, plainKey: "sk_..." }
   
2. Plain key is shown ONCE (user must copy it)
   
3. SHA256 hash of key is stored in database (DeviceApiKeys table)
   
4. Device/system uses key in X-API-Key header:
   → X-API-Key: sk_...
   
5. DeviceApiKeyResolver:
   → Computes SHA256(provided_key)
   → Looks up hash in DeviceApiKeys table
   → Returns associated DeviceId
   → Updates LastUsedAtUtc
```

### 4.3 API Key Resolution

```csharp
// DeviceApiKeyResolver.cs
public async Task<Guid?> ResolveDeviceIdFromApiKeyAsync(string? apiKey, CancellationToken ct)
{
    if (string.IsNullOrWhiteSpace(apiKey)) return null;
    
    // Compute SHA256 hash of provided key
    var hash = ComputeSha256(apiKey);
    
    // Look up hash in database
    var key = await _repository.GetByHashAsync(hash, ct);
    if (key == null) return null;
    
    // Update last used timestamp
    key.LastUsedAtUtc = DateTime.UtcNow;
    await _unitOfWork.SaveChangesAsync(ct);
    
    return key.DeviceId;
}
```

### 4.4 API Key Format

| Property | Value |
|:---------|:------|
| **Prefix** | `sk_` |
| **Key length** | 32+ characters |
| **Storage** | SHA256 hash only (64-char hex) |
| **Scope** | One key per device |
| **Revocation** | Delete via API or web portal |

### 4.5 Endpoints Using X-API-Key

| Endpoint | Description |
|:---------|:------------|
| `POST /api/webhooks/incoming` | Incoming webhook from cloud/hardware |
| `POST /api/integrations/sync` | Bulk sync data from cloud |
| `POST /api/v1/devices/{id}/heartbeat` | Device heartbeat (fallback auth) |
| `POST /api/v1/events` | Device events (fallback auth) |

## 5. User Roles & Authorization

### 5.1 Role Definitions

| Role | Value | Description | Access |
|:-----|:------|:------------|:-------|
| **Patient** | 0 | End user who takes medication | Own devices, doses, history |
| **Caregiver** | 1 | Monitors assigned patients | Patient's devices + own dashboard |
| **Admin** | 2 | Platform administrator | Full system access |

### 5.2 Authorization Model

The current MVP uses **resource-based authorization**:

```csharp
// All controllers extract userId from JWT claims
private Guid? UserId => Guid.TryParse(
    User.FindFirstValue(ClaimTypes.NameIdentifier), out var id) ? id : null;

// Handlers verify ownership before returning/modifying data
// Example: GetDevicesQueryHandler
var devices = await _deviceRepository.GetByUserIdAsync(query.UserId, ct);
// Returns only devices owned by the requesting user
```

### 5.3 Authorization Rules

| Resource | Patient | Caregiver | Admin |
|:---------|:--------|:----------|:------|
| Own devices | Read, Create, Pause, Resume | Via patient assignment | All |
| Own containers | Full CRUD | Via patient assignment | All |
| Own schedules | Full CRUD | Via patient assignment | All |
| Dispense events | Dispense, Confirm, Delay | View (via patient) | All |
| History | Own device events | Patient's device events | All |
| Notifications | Own notifications | Own + patient alerts | All |
| Travel sessions | Start/End own | View patient's | All |
| Webhooks | Own endpoints | Own endpoints | All |
| API keys | Own device keys | Own device keys | All |

### 5.4 Controller-Level Authorization

```csharp
// [Authorize] — Requires any valid JWT
[ApiController]
[Route("api/[controller]")]
[Authorize]
public class DevicesController : ControllerBase { ... }

// [AllowAnonymous] — No auth required
[HttpPost("register")]
[AllowAnonymous]
public async Task<ActionResult> Register(...) { ... }

// No [Authorize] at controller level, mixed per endpoint
[ApiController]
[Route("api/integrations")]
public class IntegrationsController : ControllerBase
{
    [Authorize]  // JWT for webhook management
    [HttpGet("webhooks")]
    public async Task<ActionResult> GetWebhooks(...) { ... }
    
    // No [Authorize] — uses X-API-Key instead
    [HttpPost("sync")]
    public async Task<ActionResult> Sync(...) { ... }
}
```

### 6.2 Production Security Checklist

| Item | Status | Action |
|:-----|:-------|:-------|
| JWT secret key | ⚠️ Change for production | Use 64+ char random key from env variable |
| HTTPS | ⚠️ Not enforced in MVP | Enable TLS via reverse proxy |
| CORS | ⚠️ AllowAny in MVP | Restrict to specific frontend domains |
| Rate limiting | ✅ Auth endpoints | Authentication endpoints are protected by a rate limiter policy ('auth') configured in Program.cs |
| Password policy | ✅ FluentValidation | Min length enforced on register |
| Token expiry | ✅ 7 days | Consider shorter for production |
| Clock skew | ✅ Zero | Strict token expiry validation |
| BCrypt hashing | ✅ Implemented | Passwords never stored in plain text |
| API key hashing | ✅ SHA256 | Plain keys never stored |

**Web Portal:**
```typescript
api.interceptors.response.use(
  (r) => r,
  (err) => {

      localStorage.removeItem('token');
      window.location.href = '/login';
    }
    return Promise.reject(err);
  }
);
```

**Mobile App:**
```typescript
api.interceptors.response.use(
  (r) => r,
  (err) => {

);
```

## 8. Token Refresh Strategy

### 8.1 Current Token Lifecycle (MVP)

| Property | Value |
|:---------|:------|
| **Token type** | Access token (JWT) |
| **Expiry** | 7 days |
| **Refresh mechanism** | None (user must re-login) |
| **Storage** | Web: localStorage, Mobile: AsyncStorage |

### 8.2 Planned: Access + Refresh Token Pattern

```
┌──────────────────────────────────────────────────────────────┐
│                TOKEN REFRESH FLOW                              │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  Login → Access Token (15 min) + Refresh Token (30 days)     │
│                                                               │
│  ┌─────────┐  Access Token   ┌──────────┐                   │
│  │  Client  │───────────────▶│  API     │  200 OK            │
│  └─────────┘                 └──────────┘                    │
│                                                               │
│  Access Token Expired:                                       │
│  ┌─────────┐  Refresh Token  ┌──────────┐                   │
│  │  Client  │───────────────▶│  API     │  New Access +      │
│  └─────────┘  POST /refresh  └──────────┘  Refresh Token     │
│                                                               │
│  Refresh Token Expired:                                      │
│  ┌─────────┐                 ┌──────────┐                   │
│  │  Client  │  Must re-login │  Login   │                   │
│  └─────────┘                 └──────────┘                    │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

### 8.3 Planned Implementation

```csharp
// New endpoint: POST /api/auth/refresh
public class RefreshTokenCommand : IRequest<AuthResponse>
{
    public string RefreshToken { get; set; }
}

public class RefreshTokenCommandHandler : IRequestHandler<RefreshTokenCommand, AuthResponse>
{
    public async Task<AuthResponse> Handle(RefreshTokenCommand request, CancellationToken ct)
    {
        // 1. Validate refresh token (check DB, not expired, not revoked)
        // 2. Generate new access token (15 min)
        // 3. Generate new refresh token (30 days) — rotate on use
        // 4. Revoke old refresh token
        // 5. Return new tokens
    }
}
```

### 8.4 Refresh Token Storage

| Property | Access Token | Refresh Token |
|:---------|:------------|:-------------|
| **Lifetime** | 15 minutes | 30 days |
| **Storage (Web)** | Memory (variable) | HttpOnly cookie |
| **Storage (Mobile)** | In-memory | Expo SecureStore (encrypted) |
| **Revocation** | Not needed (short-lived) | Database table |
| **Rotation** | On every refresh | On every refresh |

## 9. Multi-Factor Authentication (MFA) — Planned

### 9.1 MFA Strategy

| Factor | Type | Priority |
|:-------|:-----|:---------|
| **Password** | Knowledge | Current (mandatory) |
| **TOTP** | Possession (authenticator app) | Phase 2 |
| **Email OTP** | Possession (email) | Phase 2 |
| **Biometric** | Inherence (mobile only) | Phase 2 |
| **SMS OTP** | Possession (B2B only) | Phase 3 |

### 9.2 TOTP Implementation Plan

```
Setup Flow:
1. User enables MFA in Settings
2. Backend generates TOTP secret (RFC 6238)
3. Backend returns QR code (otpauth:// URI)
4. User scans with Google Authenticator / Authy
5. User enters verification code to confirm
6. Backend stores encrypted TOTP secret

Login Flow:
1. User enters email + password → 200 OK (partial)
2. Response: { requiresMfa: true, mfaToken: "temp_..." }
3. User enters 6-digit TOTP code
4. POST /api/auth/mfa/verify { mfaToken, code }
5. Backend validates TOTP code
6. Returns full access token
```

### 9.3 Database Schema (Planned)

```sql
CREATE TABLE UserMfaSettings (
    Id UUID PRIMARY KEY,
    UserId UUID NOT NULL REFERENCES Users(Id) ON DELETE CASCADE,
    MfaType VARCHAR(20) NOT NULL,  -- 'totp', 'email', 'sms'
    Secret TEXT,                     -- Encrypted TOTP secret
    IsEnabled BOOLEAN NOT NULL DEFAULT false,
    BackupCodes TEXT,               -- Encrypted JSON array of 10 codes
    CreatedAtUtc TIMESTAMP NOT NULL,
    LastUsedAtUtc TIMESTAMP
);
```

## 10. Password Reset Flow — Planned

### 10.1 Password Reset Sequence

```
User                                Server                          Email
  │                                   │                              │
  │ POST /api/auth/forgot-password   │                              │
  │ { email }                         │                              │
  │──────────────────────────────────▶│                              │
  │                                   │                              │
  │                                   │ 1. Find user by email       │
  │                                   │ 2. Generate reset token     │
  │                                   │    (random 64 bytes, SHA256 │
  │                                   │    stored, 1 hour expiry)   │
  │                                   │ 3. Send email               │
  │                                   │──────────────────────────────▶│
  │                                   │                              │
  │ 200 OK (always, prevent enum)    │                              │
  │◀──────────────────────────────────│                              │
  │                                   │                              │
  │ User clicks email link            │                              │
  │ /reset-password?token=xxx         │                              │
  │                                   │                              │
  │ POST /api/auth/reset-password    │                              │
  │ { token, newPassword }            │                              │
  │──────────────────────────────────▶│                              │
  │                                   │                              │
  │                                   │ 4. Validate token (hash,    │
  │                                   │    not expired, not used)    │
  │                                   │ 5. Hash new password        │
  │                                   │ 6. Update user              │
  │                                   │ 7. Invalidate all sessions  │
  │                                   │ 8. Mark token as used       │
  │                                   │                              │
  │ 200 OK                            │                              │
  │◀──────────────────────────────────│                              │
```

### 10.2 Security Requirements

| Requirement | Implementation |
|:-----------|:-------------|
| Token entropy | 64 random bytes (Base64URL encoded) |
| Token storage | SHA256 hash only (never store plain) |
| Token expiry | 1 hour |
| Token usage | Single use (invalidated after reset) |
| Rate limiting | 3 requests per 15 min per email |
| Response | Always 200 OK (prevent email enumeration) |
| Password rules | Min 8 chars, 1 uppercase, 1 number |
| Session invalidation | Revoke all refresh tokens on reset |

## 11. Device Authentication (Firmware)

### 11.1 Device Registration Authentication

```
┌──────────────────────────────────────────────────────────────┐
│              DEVICE AUTHENTICATION FLOW                        │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  First Boot (Provisioning):                                  │
│  1. Device sends MAC + hardware ID                           │
│  2. Server creates device record                             │
│  3. Server returns device JWT (RS256, 1 year)                │
│  4. Device stores JWT in NVS (encrypted flash)               │
│                                                               │
│  Runtime:                                                    │
│  1. Device uses JWT for all API calls                        │
│  2. Heartbeat response can include token refresh command     │
│  3. If 401 → device re-registers                            │
│                                                               │
│  Production (Planned):                                       │
│  1. Device has X.509 client certificate (Secure Element)     │
│  2. Mutual TLS authentication                                │
│  3. Certificate rotation via OTA                             │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

### 11.2 Device JWT vs User JWT

| Property | User JWT | Device JWT |
|:---------|:---------|:-----------|
| **Algorithm** | HS256 | RS256 (planned) |
| **Expiry** | 7 days (→ 15 min) | 1 year |
| **Claims** | userId, email, role | deviceId, deviceType |
| **Storage** | localStorage / memory | NVS (encrypted flash) |
| **Refresh** | Refresh token | Heartbeat command |
| **Revocation** | On logout/password change | Remote wipe command |

### 13.2 Data Access Rules

| Rule | Implementation |
|:-----|:-------------|
| **Ownership check** | Handler queries data filtered by `userId` from JWT claims |
| **Caregiver access** | Handler checks if requesting user is caregiver of the resource owner |
| **Admin bypass** | Admin role skips ownership checks (planned) |
| **Device scope** | Device API resolves device from token/API key, no cross-device access |
| **Read vs Write** | Caregivers have read access to patient data, no write access |
